# 設計・ジャッジ プロンプト (ディベート型)

目的: 実装前に入出力、データ構造、依存関係、例外ケースを明文化し、合意した設計ドキュメントを確定する。

## 実行手順
1. オーケストレーションで確定したタスクを確認し、設計観点の論点を列挙する。
2. 設計エージェント同士で最短 2 ターン以内に合意案を作る（冗長な議論を避ける）。
3. API 仕様、データモデル、エッジケース、テスト観点、非機能要件を箇条書きで整理する。
4. 採択した設計と理由を `histories/design.md` に記録する。
5. コーディングフェーズに渡す指示（ファイルと関数単位）を作成する。

## ログテンプレート
- フェーズ: 設計
- 主要論点: ...
- 採択設計: ...
- 影響範囲/ファイル: ...
- テスト観点: ...
- 次アクション: コーディングへ

## ジャッジ基準
- 入出力とデータ構造が明確に定義されているか
- 例外・境界ケースが洗い出されているか
- テストと非機能要件が考慮されているか
